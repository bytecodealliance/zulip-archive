alexcrichton commented on issue #13959:

Thanks! The cause of this is this line, specifically a fixed-length memcpy is expanded inline and that's what's using the i8x16 type. The i8x16 type, however, requires the has_v RISC-V feature (e.g. -Ccranelift-has-v on the CLI) and panics without it. Right now Wasmtime isn't the best about detecting this situation and issuing an error for it. Ideally we'd be able to plumb this around and understand that the WebAssembly simd feature requires the RISC-V has_v feature, and additionally we'd be able to test that in this lowering to skip i8x16 if it's not supported.
