This is the first part of #13707

The goal is to cut down on the amount of manual boilerplate involved in defining VMFoo types in Wasmtime. Right now, they are effectively defined repeatedly in multiple places:

  • The actual type definition used in the runtime, in crates/wasmtime/src/runtime/vm/vmcontext.rs

  • The type's offsets (parameterised over a target pointer size to support cross compilation) in crates/environ/src/vmoffsets.rs

  • The VmType variant and all the helpers we define on AliasRegion to load and store to that field with the correct alias region, in crates/cranelift/src/alias_region.rs

And these all have to agree with each other, or else we can get miscompiles and potential vulnerabilities.

Instead of repeating ourselves across those locations, this commit introduces a for_each_vm_type! macro-rules macro that takes another macro as an argument and invokes it once, passing in each of the VMFoo types' definitions.

While we will eventually move most of the VMFoo types into the macro, to keep this commit relatively small, it starts with just four simple types:

  • VMMemoryDefinition
  • VMTableDefinition
  • VMGlobalDefinition
  • VMTagDefinition

The expectation is that we will add more types and expand the macro more as necessary in follow up commits.

We use the for_each_vm_type! macro in three places:

  1. In vmoffsets.rs we use it to define the size, alignment, and field offsets for each VMFoo type. The generated code is slightly different from the old code, so code that used to be, for example, ptr.vmmemory_definition_base() is now ptr.vm_memory_definition().base().

  2. In vmcontext.rs we use it to define the VMFoo types themselves, along with tests to assert that the types's actual size, alignment, and field offsets match what we expected in (1).

  3. In alias_regions.rs we use it to define AliasRegion helper methods to load and store to these fields with the correct alias regions. Again, the generated code here is slightly different from the old code, and code that used to be, for example, self.alias_regions.vmmemory_definition_current_length(...) becomes self.alias_regions.vm_memory_definition().current_length().load(...).

Overall, this doesn't save on lines of code yet, but it is expected to once we port the rest of the VMFoo types over to the macro.
