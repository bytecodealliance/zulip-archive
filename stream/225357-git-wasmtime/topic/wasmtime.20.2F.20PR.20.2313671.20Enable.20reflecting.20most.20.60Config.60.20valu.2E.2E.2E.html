This commit adds a number of pieces of plumbing/etc where the end goal is to be able to reflect upon the configuration values from an Engine. The pieces added here are:

  • Many new get_* methods now exist on Engine to read out the configuration values specified. These are intended to not return an Option<T> in most cases to accurately reflect Wasmtime's defaults and configuration.

  • The wasmtime-cli-flags is now feature'd to have a separate clap, serde, and toml feature.

  • The wasmtime-cli-flags crate structures now sport a Serialize implementation to go along with their Deserialize implementation.

  • A new wasmtime_cli_flags::CommonOptions::from_engine method was added to extract configuration from an Engine.

The intent here is that all raw getters are present on Engine itself, not Config. This is an intentional decision as Config is a part-way piece of configuration which doesn't necessarily have all defaults/etc loaded to determine the true final configuration options. An Engine, however, is a finalized piece of configuration which has a finalized Tunables, for example.

Additionally Engine has accessors based on Config names, but it's also easy to forget to add these over time. Additionally everything's not in a nicely serializable format, but that's what CommonOptions packages up nicely. The from_engine method bridges this gap by connecting all Engine options to an exhaustive construction of all CLI options. This means that if we forget to add one in the future we'll at least get a compiler error to alert us.

The overall goal with this is to assist with embeddings updating wasmtime over time. It should now be much easier to serialize Wasmtime's default configuration to a file and persist that across major Wasmtime versions. This enables embeddings to configure modules uniformly over time and have at least mostly similar settings across versions of Wasmtime.

I'll note as well that from_engine is not exhaustive. There are configuration options that fundamentally can't be captured in a serializable format, such a MemoryCreator (a trait object). This is documented on from_engine where it's intended to represent a major subset of configuration but not exhaustive. This is suitable for applications, though, which exclusively derive a Config from CommonOptions (or similar) where it can be understood that everything is still serializable there.
