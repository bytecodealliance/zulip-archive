Implements the core reference type instructions for Winch (funcref only):

  • ref.null: pushes a null funcref (pointer-sized zero)
  • ref.is_null: compares a ref to zero, produces i32 0/1
  • ref.func: calls the existing ref_func builtin to get a funcref for a function index
  • typed select: delegates to the existing untyped select implementation

Non-func heap types (externref, etc.) bail with an unsupported error, consistent with how table_get, table_set, and other table ops already handle them.

Relates to #8088, #9924.

visit_ref_is_null hardcodes RegImm::i64(0) and OperandSize::S64. Since ref.is_null is defined over pointer-sized references and this module already threads env.ptr_type() elsewhere (e.g. visit_ref_null, memory_grow adjustments), it would be more robust to derive the operand size/immediate from the target pointer type (or from the popped value’s WasmValType) rather than assuming 64-bit.

This avoids future miscompiles if/when Winch gains 32-bit pointer support or if pointer width becomes configurable in more places.

            let (zero, size) = match self.env.ptr_type() {
                WasmValType::I64 => (RegImm::i64(0), OperandSize::S64),
                WasmValType::I32 => (RegImm::i32(0), OperandSize::S32),
                _ => bail!(CodeGenError::unsupported_wasm_type()),
            };
            masm.cmp_with_set(writable!(reg), zero, IntCmpKind::Eq, size)?;

Copilot created PR review comment:

visit_typed_select ignores the immediate result type and delegates to visit_select, which pushes val2 with its original WasmValType. For typed select, operands may have different (e.g. different ref nullability/subtypes) types while the immediate type defines the result; keeping val2's type can leave the value stack with an incorrect type for subsequent lowering decisions.

Consider converting the ValType immediate to the corresponding WasmValType (e.g. via the existing TypeConverter) and updating the selected value's tracked type to that result type (and/or asserting it matches the operand widths) before pushing it back onto the stack.

    fn visit_typed_select(&mut self, ty: ValType) -> Self::Output {
        let cond = self.context.pop_to_reg(self.masm, None)?;
        let val2 = self.context.pop_to_reg(self.masm, None)?;
        let val1 = self.context.pop_to_reg(self.masm, None)?;

        // Determine the result machine type from the immediate. Numeric types
        // map directly; all ref-like types are represented using the pointer
        // type in this backend.
        let result_ty = match ty {
            ValType::I32 => WasmValType::I32,
            ValType::I64 => WasmValType::I64,
            ValType::F32 => WasmValType::F32,
            ValType::F64 => WasmValType::F64,
            ValType::V128 => WasmValType::V128,
            _ => self.env.ptr_type(),
        };

        // In well-typed Wasm, the operands to a select have the same
        // representation type as the result. Assert this in debug builds.
        debug_assert_eq!(val1.ty, val2.ty);
        debug_assert_eq!(val1.ty, result_ty);

        self.masm
            .cmp(cond.reg, RegImm::i32(0), OperandSize::S32)?;
        // Conditionally move val1 to val2 if the comparison is not zero.
        self.masm.cmov(
            writable!(val2.into()),
            val1.into(),
            IntCmpKind::Ne,
            val1.ty.try_into()?,
        )?;

        // Push the selected value back on the stack using the result type
        // derived from the immediate, not the original type of `val2`.
        self.context.stack.push(Val::reg(val2.reg, result_ty));

        self.context.free_reg(val1.reg);
        self.context.free_reg(cond);

        Ok(())
